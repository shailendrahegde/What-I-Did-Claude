RULE 1 — GOAL GROUPING (most important rule)
═══════════════════════════════════════════

Group work into BUSINESS GOALS. A goal = the high-level outcome accomplished, not the technical steps.

IRON RULE: Everything in the same session that touches the same system is ONE GOAL.
Iterating, refining, fixing bugs, adding features — all tasks within the original goal, not separate goals.

DEFAULT RULE: When in doubt, MERGE. Too few goals is better than too many.

The only valid reason to create a new goal is: work on a COMPLETELY DIFFERENT subject in a DIFFERENT
project/session with ZERO shared files or dependencies.

ROOT/HOME FOLDER EXCEPTION: If a session's PROJECT is a user home directory (e.g. "jsmith",
"username", or a path like /Users/name or C:\Users\name), the user was likely doing ad-hoc work
that does NOT belong to a single project. In this case:
- Do NOT merge all sessions from the same home folder into one giant goal.
- Only group sessions together if they share CLEAR evidence of the same project: same files
  modified, same topic/domain, same repository, or explicit continuation of prior work.
- Separate ad-hoc tasks (e.g. a quick research question, installing a tool, answering a question)
  should each be their own small goal or grouped only with closely related tasks.
- Look at the actual CONTENT and FILES in each session, not just the project path.

GOOD GOAL TITLES (business outcome, verb-first, INCLUDE the project/repo/tool name):
  "Built a daily Claude activity analytics tool with branded HTML reports"
  "Shipped the whatidid reporting tool from concept to working system"
  "Provided strategic rewrite recommendations for the DPG-AI-Skilling presentation"
  "Diagnosed and resolved checkout regression in the Contoso API"

BAD GOAL TITLES (too granular, or missing project context):
  "Set up authentication"        ← part of a larger goal

  "Updated README"               ← which README? Include the repo/project name
  "Initialized git repository"   ← setup step, not the goal itself
  "Improved documentation"       ← too vague — documentation for what?

TITLE RULE: The goal title must describe the PRIMARY DELIVERABLE, not the first thing done.
If a session starts with "prepare for github" but spends 80% of time building a report tool,
the title should be about the report tool, not the git setup.

RULE 3 — EFFORT ESTIMATES (research-calibrated)
═══════════════════════════════════════════

human_hours = what a skilled professional would need WITHOUT AI assistance.
Estimate realistically — what would an expert actually bill for this work?

RESEARCH CONTEXT (Alaswad et al. 2026, Cambon et al. 2023, Ziegler et al. 2024):
- LLM-assisted work completes 1.4–4× faster than unassisted work (Microsoft studies)
- 78% of tasks humans rated "high complexity" completed with <25% of human effort using LLM
- BUT the human EQUIVALENT is what matters: reverse the speedup to estimate what it would take WITHOUT AI
- No single metric captures effort — use MULTIPLE signals together (SPACE framework, Forsgren 2021)
- Iteration count and oversight effort are the strongest new predictors of true complexity

Each session includes a SIGNALS block. USE IT to calibrate — do not ignore these numbers.

STEP 1 — Classify the task type using the SIGNALS block:
  Mechanical (install, deploy, git push, config)    → 0.25-0.5h regardless of other signals
  Research/Analysis (read-heavy tools, low edits)   → estimate from conversation turns × 5-10 min
  Implementation (edit-heavy, lines added)           → estimate from lines ÷ 125 LoC/hr + research overhead
  Debugging (run-heavy, high iteration depth)        → estimate from base + 50% debugging premium
  Complex multi-step (balanced reads+edits+runs)     → use highest signal as base, apply multipliers

STEP 2 — Compute base estimate from the STRONGEST primary signal:
  Conversation turns (PRIMARY interaction signal — only SUBSTANTIVE turns count):
    Trivial turns ('yes', 'commit', 'looks good', short confirmations) are excluded.
    Each substantive turn ≈ 5-7 min of human thinking.
    1-3 = 0.25h, 4-8 = 0.75h, 9-15 = 1.5h, 16-30 = 3h, 31-60 = 5h, 61-100 = 8h, 100+ = 10h
  Tool invocations (WEIGHTED by action type — expert is more deliberate than AI):
    The SIGNALS block shows the breakdown: reads, edits, runs.
    reads (file views, searches) ≈ 0.3 min each — quick scan
    edits (file edits, creates) ≈ 1.5 min each — expert makes fewer, targeted edits
    runs (commands, tests) ≈ 0.75 min each — execution + glance at result
    other (agent overhead) ≈ 0 min — not human work
    Example: 200 tools (60 reads, 40 edits, 30 runs, 70 other) =
      60×0.3 + 40×1.5 + 30×0.75 = 18+60+22.5 = 100.5 min = 1.7h
  Active engagement time: active_minutes × 3 (midpoint of 1.4–4× research range, Cambon 2023)
  Lines of code (ADDITIVE — added on top of the base, not part of max):
    Expert writes 100-150 LoC/hr. 50-150 lines = 0.75h, 150-300 = 1.5h, 300-500 = 2.5h, 500+ = 4h+

  base = max(turns_h, tools_h, active_h)
  total = (base × complexity multipliers, capped at 2.2×) + lines_h

STEP 3 — Apply complexity multipliers from the SIGNALS block:
  These reflect research findings that iteration, scope, and rework can MORE THAN DOUBLE
  the base effort (Alaswad 2026: 22% of tasks took >180% expected effort):

  Conversation turns > 15: +15% (Chen 2023: ambiguous tasks → lengthy dialogues)
  Conversation turns > 40: +20% more (genuinely hard iterative problem-solving)
  Iteration depth > 5 edits/file: +15% (Alaswad 2026: iterative reasoning cycles)
  Iteration depth > 12 edits/file: +20% more (extensive debugging/rework)
  Files touched > 3: +10% (Morcov 2020: integration overhead starts early)
  Files touched > 10: +20% more (Tregubov 2017: 17% time lost to context-switching)
  Engagement ratio < 10%: task was thinking-heavy — active_minutes understates true effort
  Lines removed > 30% of lines added: refactoring work, add 10-20%

  Maximum combined multiplier: ~2.2× — a heavily iterative, broad-scope, debugging-heavy
  task can more than double the base estimate.

STEP 4 — Reality-check against anchors:
  0.25h  — Trivial: install a package, run a CLI command, toggle a config, copy files, push to git
  0.5h   — Simple: minor code edit, format/style tweak, rename, answer a question

  1.0-1.5h — Moderate: implement a small feature, debug an unknown issue, draft a short document

  8-16h  — Large: build a full system from scratch, multi-day design-implement-test cycle
